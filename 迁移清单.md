# MuJoCo 新版迁移清单（`mujoco-py` -> `mujoco`）

## 1. 目标与范围
- 目标：把当前项目从旧版 MuJoCo 依赖（通常是 `mujoco-py` + 本地二进制）迁移到 `pip install mujoco` 可用的新栈。
- 范围：环境创建、仿真调用、观测/动作、并行采样、渲染录像、训练脚本与依赖配置。
- 非目标：本阶段不改 PPO/SNN 算法逻辑，仅做运行时兼容与行为等价。

## 2. 建议迁移策略（分阶段）
- 阶段 A（兼容层先行）：
  - 先引入新依赖并做最小兼容层，让代码同时可在“旧路径/新路径”下运行。
  - 保持训练脚本入口和配置字段尽量不变。
- 阶段 B（彻底切换）：
  - 去掉旧 MuJoCo 依赖路径与分支，统一到新 `mujoco` API。
  - 清理兼容代码，更新文档与 Docker。

## 3. 文件级改造清单

### 3.1 核心环境与 MuJoCo 调用
- [ ] `metamorph/envs/tasks/unimal.py`
  - 检查是否直接访问 `sim/model/data` 或 `mjlib` 风格接口。
  - 替换为新 `mujoco` 对应 API（状态读取、step、reset、关节与刚体数据访问）。
- [ ] `metamorph/utils/mjpy.py`
  - 该文件大概率是旧 `mujoco-py` 适配层，优先改造成新 API 封装。
  - 建议把“模型加载、仿真推进、数据索引访问”统一收口到这里，减少业务文件改动面。
- [ ] `metamorph/utils/camera.py`
  - 检查渲染与相机读取接口是否依赖旧 viewer/context。
  - 新版 `mujoco` 渲染接口与上下文管理方式不同，需要单独适配。
- [ ] `metamorph/envs/modules/agent.py`
  - 检查 XML 加载后对 model/data 的字段访问是否有 API 差异。

### 3.2 Gym / Gymnasium 适配层
- [ ] `metamorph/envs/__init__.py`
  - 注册逻辑确认：`gym` 与 `gymnasium` 的 registry/entry point 行为差异。
- [ ] `metamorph/algos/ppo/envs.py`
  - `make_env`、`RecordEpisodeStatistics`、`TimeLimitMask` 要兼容新 step/reset 返回值。
  - 重点：`reset()` 是否返回 `(obs, info)`，`step()` 是否返回 `(obs, reward, terminated, truncated, info)`。
- [ ] `metamorph/envs/vec_env/*`
  - 尤其 `subproc_vec_env.py`、`vec_normalize.py`、`pytorch_vec_env.py`：
  - 确保 done 语义在 terminated/truncated 拆分后不被破坏（PPO bootstrapping 依赖这里）。

### 3.3 训练与评估入口
- [ ] `tools/train_ppo.py`
  - 检查初始化和异常处理在新 API 下是否需要额外字段。
- [ ] `tools/generate_video.py`（及同类脚本）
  - 录像路径依赖渲染接口，通常需要同步适配。

### 3.4 配置与部署
- [ ] `docker/build_files/requirements.txt`
  - 替换旧 MuJoCo 相关依赖，加入 `mujoco` 与兼容版本的 `gymnasium`/桥接包。
- [ ] `docker/build_files/install_gpu_deps.sh`
  - 删除旧版 MuJoCo 二进制下载/环境变量设置步骤（如有）。
- [ ] `docker/Dockerfile`
  - 依据新版依赖更新系统包。
- [ ] `README.md`
  - 更新安装步骤，明确不再需要手动下载旧 MuJoCo 文件。

## 4. API 差异重点核查项（高风险）
- [ ] reset/step 返回值长度变化是否影响所有 wrapper 和 vecenv。
- [ ] done vs terminated/truncated 映射是否正确影响 GAE/timeout 逻辑。
- [ ] 模型字段索引方式差异（关节、刚体、geom 名称到 id 的映射）。
- [ ] 渲染线程/上下文在多进程 vec env 下是否稳定。
- [ ] XML 载入与默认参数行为差异是否改变动力学（reward 曲线可能漂移）。

## 5. 验证计划（每阶段都执行）
- [ ] 单环境 smoke test：`env.reset()`、`env.step()`、`obs shape`、`reward` 正常。
- [ ] 向量环境 smoke test：`DummyVecEnv` 与 `SubprocVecEnv` 均可跑 100~500 steps。
- [ ] PPO 短训测试：`tools/train_ppo.py --cfg ./configs/ft.yaml` 小步数跑通。
- [ ] 录像测试：`tools/generate_video.py` 可输出视频。
- [ ] 回归对比：同种子下旧版 vs 新版 reward 趋势、FPS、稳定性。

## 6. 推荐执行顺序（实操）
1. 先做依赖与最小环境启动（只要求 reset/step 通）。
2. 再改 vec env 与 wrapper（保证训练数据链路稳定）。
3. 再打通 PPO 短训与日志。
4. 最后处理渲染/录像与文档清理。

## 7. 预估工作量
- 最小可运行迁移：1~3 天（取决于 `mjpy.py` 耦合程度）。
- 稳定可复现实验迁移：3~7 天（含回归和参数微调）。

## 8. 交付标准
- [ ] 全新环境下仅通过 `pip` 安装依赖即可启动训练（无需手工下载旧 MuJoCo 文件）。
- [ ] `configs/ft.yaml` 能完成至少一次短训并输出 checkpoint。
- [ ] 视频生成与评估脚本可正常运行。
- [ ] README 安装文档与实际流程一致。
